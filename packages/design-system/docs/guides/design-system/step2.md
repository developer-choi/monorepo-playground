## 1. 문제: 겹치는데 따로 만든 컴포넌트

step1에서 공통 컴포넌트로 묶고 나니, 이번엔 비슷한 UI가 추가될 때 문제가 드러났습니다. Dialog와 Bottom Sheet는 배경을 어둡게 덮는 처리, 배경을 눌렀을 때 닫히는 동작처럼 겹치는 부분이 많았습니다.

그런데도 둘을 따로 만들어 두니, 그 겹치는 부분을 같이 쓸 수 없었습니다. 한쪽을 고치면 다른 쪽도 똑같이 고쳐야 했습니다.

## 2. 원인: 결과만 보고 설계는 보지 않음

step1에서 MUI를 참고했다고 했지만, 참고한 건 공식 문서에 적힌 스펙, 즉 겉으로 드러난 결과물뿐이었습니다. 그 컴포넌트가 내부적으로 어떤 하위 컴포넌트로 나뉘는지, 즉 설계까지는 보지 않았습니다.

그래서 만들어진 컴포넌트는 하나의 덩어리였습니다. 배경 처리와 닫기 동작이 Dialog 안에 박혀 있어, 다른 컴포넌트가 같은 동작을 쓰려면 복사하는 길밖에 없었습니다.

## 3. 해법: 원본 코드의 계층 구조 분석

이번엔 MUI 원본 코드를 직접 분석했습니다. Dialog 하나가 사실은 여러 하위 컴포넌트가 쌓인 구조라는 걸 그제야 알았습니다.

```
Dialog
└─ Modal          // 열고 닫기, 스크롤 잠금
   ├─ Portal      // DOM 최상단에 렌더
   ├─ Backdrop    // 배경 어둡게, 클릭 시 닫힘
   └─ FocusTrap   // 포커스를 모달 안에 가둠
```

겹치던 부분은 사실 Modal이라는 공통 컴포넌트였습니다. 이 Modal을 분리하니, Dialog와 Bottom Sheet가 같은 토대를 공유하게 됐습니다.

```tsx
// before: 배경·닫기·포커스가 컴포넌트마다 박혀 중복
function Dialog() { /* 배경 + 닫기 + 포커스 + 내용 */ }
function BottomSheet() { /* 위와 똑같은 처리를 다시 작성 */ }

// after: 공통 동작을 Modal로 분리
function Dialog() { return <Modal>{/* Dialog 내용 */}</Modal>; }
function BottomSheet() { return <Modal>{/* Bottom Sheet 내용 */}</Modal>; }
```

## 4. 추상화 판단: 디자이너와의 합의

분석은 중복을 줄여줬을 뿐 아니라, 디자이너에게 미리 물어볼 거리도 만들어줬습니다. 원본에서 Backdrop이라는 공통 컴포넌트의 존재를 알게 되니, "앞으로 Bottom Sheet 같은 걸 만들면 그 배경도 Dialog 배경과 같은 스타일로 갈 건가요?" 같은 질문을 먼저 던질 수 있었습니다.

step1에서는 제 예측이 표준 도입을 이끌었다면, 여기서는 코드 분석이 질문을 끄집어낸 셈입니다.

다만 모든 공통 컴포넌트를 추상화하진 않았습니다. 추상화 여부는 "이 컴포넌트가 앞으로도 함께 갈 것인가"를 디자이너와 합의해 정했습니다.

- **Modal (추상화함)**: Dialog와 Bottom Sheet를 모두 Modal로 볼 수 있냐 물었더니 그렇다고 하셔서, Modal과 Backdrop을 공통 컴포넌트로 분리했습니다.
- **ButtonBase (추상화 안 함)**: 모든 클릭 요소에 공통 애니메이션이 생길 수 있냐 물었더니 아니라고 하셔서, Button 아래 공통 컴포넌트는 두지 않았습니다.

## 5. 단점: 원본 분석에 드는 시간

원본 코드를 읽고 계층을 파악하는 데는 시간이 꽤 들었습니다. 공식 문서만 보던 step1보다 분명히 느린 작업이었습니다.

이 비용은 AI로 상쇄했습니다. 코드를 통째로 읽힌 뒤 "이 컴포넌트는 어떤 하위 컴포넌트로 나뉘나"를 물으면, 분석의 출발점을 빠르게 잡을 수 있었습니다.

## 결과

겹치던 동작을 공통 컴포넌트로 분리하고 나니, 비슷한 UI 계열이 늘어도 중복 없이 같은 토대를 재사용할 수 있었습니다.

새 UI를 만들 때도 처음부터 다시 짜는 대신 어떤 공통 컴포넌트 위에 얹을지부터 생각하게 됐습니다.
