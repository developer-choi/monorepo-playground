## 문제: 페이지마다 따로 만든 컴포넌트

페이지마다 버튼과 모달을 그때그때 새로 작성하다 보니, 같은 UI가 여러 벌로 존재했습니다. 버튼 스타일도 미묘하게 달라, 같은 서비스인데도 화면마다 버튼이 조금씩 다르게 생겼습니다.

## 첫 번째 답: 공통 컴포넌트

자연스러운 답은 공통 컴포넌트였습니다.

- **목적**: 한 번 만들어 여러 페이지에서 재사용합니다.
- **전제**: 한 곳에서만 쓰는 컴포넌트라면 그 페이지의 요구만 맞추면 됩니다. 하지만 여러 페이지가 공유하는 순간, 그 모든 페이지의 요구를 동시에 만족하는 스펙이어야 합니다.

## 예측할 수 없는 미래 요구

Figma를 그대로 개발로 옮기는 것도 방법이었습니다. 하지만 그렇게만 하면 머지않아 이런 일이 생길 것이라 예상했습니다.

- 버튼을 로딩 상태 없이 만들어 두면, 곧 "로딩도 필요하다"는 요구가 올 것입니다
- "삭제처럼 파괴적인 동작은 배경색을 다르게" 같은 요구가 붙을 것입니다
- 강조하는 버튼에 쓸 primary 색, 취소처럼 덜 중요한 버튼에 쓸 회색처럼 자주 쓰는 색을 지정해야 할 것입니다

요구가 추가될 때마다 컴포넌트를 다시 고치고, 그걸 쓰던 기존 페이지까지 되돌아봐야 합니다.

## 해법: 검증된 표준 참고 (Material Design)

표준을 직접 새로 정의하는 대신, 이미 검증된 표준을 참고하기로 했습니다. 그게 Material Design이었고, MUI 공식 문서의 스펙을 참고했습니다.

"우리 서비스의 미래도 결국 저들이 정리해 둔 스펙과 크게 다르지 않을 것"이라는 판단이었습니다.

다만 MUI 컴포넌트 라이브러리를 설치해서 그대로 가져다 쓴 건 아닙니다. 어떤 props를 받고 어떤 기능을 지원하는지를 참고해 직접 만들었습니다.

이미 스타일이 입혀진 라이브러리를 우리 브랜드에 맞게 바꾸려면, 깔아 둔 스타일을 하나하나 덮어써야 합니다. 덮어쓰기는 CSS 우선순위 충돌이 됩니다.

당시 디자인 시스템이 복잡하지 않아, 덮어쓰는 비용보다 직접 만드는 편이 더 저렴했습니다.

다만 이 방식에도 한계가 있었고, 그 한계는 이후 단계에서 보완합니다.

## 당시 놓친 대안: Headless UI

하지만 이때는 headless라는 개념 자체를 몰라, 컴포넌트를 직접 개발하는 길을 택했습니다.

> 직접 개발은 접근성·키보드·포커스 같은 동작까지 전부 직접 만들어야 해서, 제대로 갖출수록 품이 많이 드는 한계를 남겼습니다. 이 한계는 [Step 3](./step3.md)에서 Radix 기반으로 다시 설계하며 극복합니다.
