## 문제: 직접 만들기의 부담

표준으로 묶고, 구조를 정리했습니다. 그런데도 바뀌지 않은 게 하나 있었습니다.

결국 그 컴포넌트를 우리가 직접 만들어야 한다는 점입니다.

모달 하나만 봐도 열고 닫기, 바깥을 눌렀을 때 닫기, Escape로 닫기, 열려 있는 동안 포커스를 안에 가두기, 뒤 배경 스크롤 잠그기까지 챙겨야 합니다. 키보드만으로 쓰는 경우까지 고려하면 직접 짜야 할 동작이 계속 늘었습니다.

## 전환: 동작을 직접 만들지 않는 headless

이 무렵 headless라는 개념을 알게 됐습니다. 동작은 라이브러리가 제공하고 시각 스타일은 전혀 입히지 않은 채 넘겨주는 방식입니다.

[step1](./step1.md)에서는 스타일이 입혀진 라이브러리를 우리 브랜드에 맞게 바꾸는 비용이 크다고 봤습니다. headless는 스타일이 없으니, 그 비용 없이 동작만 가져올 수 있습니다.

그래서 이번엔 라이브러리를 쓰기로 했고, Radix를 골랐습니다.

Radix primitive는 동작만 담겨 있고 시각 스타일은 없습니다. 그래서 동작은 Radix에 맡기고, 모양은 우리 클래스로 입혔습니다.

```tsx
// before: 동작을 전부 직접 구현
function Modal({open, onClose, children}) {
  // 열고 닫기, 바깥 클릭 감지, Escape 키 처리,
  // 포커스 가두기, 배경 스크롤 잠금 ... 전부 우리 몫
}

// after: 동작은 Radix가, 시각은 className으로만
function Dialog({open, onClose, children}) {
  return (
    <RadixDialog.Root open={open} onOpenChange={(isOpen) => !isOpen && onClose()}>
      <RadixDialog.Portal>
        <RadixDialog.Overlay className={styles.overlay} />
        <RadixDialog.Content className={styles.paper}>{children}</RadixDialog.Content>
      </RadixDialog.Portal>
    </RadixDialog.Root>
  );
}
```

포커스 가두기, 바깥 클릭, Escape, 배경 스크롤 잠금은 이제 Radix가 처리합니다. 우리는 Overlay와 Content에 클래스로 모양만 입혔습니다.

버튼도 마찬가지로, 버튼처럼 보이는 링크가 필요할 때 Radix의 Slot으로 동작을 자식 엘리먼트에 얹었습니다.

## 세 갈래 비교: 직접 / styled 라이브러리 / headless

Radix 같은 headless 라이브러리는 여러 서비스가 오래 써오며 다양한 환경에서 검증한 동작을 그대로 가져다 쓰는 셈입니다. 직접 짠 동작과 달리, 이미 수많은 환경에서 굴러본 코드입니다.

직접 만들던 시절 가장 힘들었던 건 코드를 짜는 시간이 아니라 다 만든 다음이었습니다. iOS, 안드로이드, 크롬을 비롯한 모던 브라우저마다 간혹 동작이 어긋났고, 그게 어디서 왜 생기는지 재현하고 디버깅하는 일이 특히 괴로웠습니다.

내 코드는 멀쩡해 보이는데 특정 환경에서만 틀어지니, 원인을 좁히는 데만 한참이 걸렸습니다. headless로 옮기고 나서는 이 부류의 문제를 직접 떠안지 않게 됐습니다.

그래서 세 가지 방식을 비교하면 이렇습니다.

- **직접 구현**: 모든 걸 우리 것으로 쥐지만, 위에서 말한 환경별 동작 문제를 전부 직접 책임집니다.
- **styled 라이브러리(MUI류)**: 모양까지 입혀져 즉시 쓸 수 있지만, 그 라이브러리의 시각 정체성에 묶입니다.
- **headless(Radix)**: 동작은 검증된 라이브러리에 맡기고, 시각은 우리가 쥡니다.

직접 구현과 비교하면, headless가 상위호환이었습니다. 직접 구현의 장점인 제어권은 headless도 시각을 우리 코드에 그대로 두므로 잃지 않고, 동작을 직접 책임지는 부담만 덜어냅니다.

이 프로젝트에서 직접 구현을 택할 이유는 거의 없었습니다.

styled 라이브러리와 비교하면 무조건은 아닙니다. 빠른 출시가 전부인 MVP였다면, 모양까지 입혀져 바로 쓸 수 있는 쪽이 나았을 겁니다.

하지만 우리는 디자이너와 함께 프로젝트마다 시각을 바꿔 끼우는 재사용 시스템을 만드는 상황이었습니다.

이미 스타일이 입혀진 라이브러리를 우리 브랜드에 맞게 바꾸려면, 깔아 둔 스타일을 덮어써야 합니다. 덮어쓰기는 CSS 우선순위 충돌이라, 선택자를 키우거나 강제 지정을 쓰게 되고, 버전업 때 같이 깨지기 쉽습니다.

headless는 스타일이 없으니, 덮어쓸 것 자체가 없습니다. 처음부터 우리 브랜드를 그리면 됩니다.

그래서 브랜드가 라이브러리 기본 룩과 다를수록, 덮어쓰는 쪽보다 처음부터 그리는 쪽이 비용이 작았습니다. 프로젝트마다 스타일을 교체하는 우리로선 이 차이를 매번 치르는 셈이라, 격차는 더 벌어졌습니다.

이 조건에서는 동작은 라이브러리에 맡기고 시각은 우리가 쥐는 headless가 상위호환이었습니다.

## 결과

headless로 옮기고 나서, 동작을 직접 책임지던 부담과 환경마다 터지던 버그에서 벗어났습니다.

시각은 여전히 우리 손에 있으니, 디자인을 바꿔도 우리 클래스만 손보면 됐습니다.
