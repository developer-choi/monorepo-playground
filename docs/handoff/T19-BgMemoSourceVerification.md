# T19. BG 산출물 메모를 1차 소스로 검증하지 않고 PR 결정에 직행한 사고

## 작성 맥락

이 문서는 **채용과제 프로젝트(`assignment-playground`)에서 PR_1_PLAN 진행 중 발생한 사고**의 회고를 MP로 환류한 것이다. 채용과제 레포는 폐기 예정이므로, 이 사고에서 도출되는 룰·검증 패턴은 MP·AC(`ai-contexts`)에 남겨야 다음 채용과제·사이드 프로젝트에서 재발 방지가 된다.

### 사고가 발생한 환경

- 프로젝트: `assignment-playground` (랭디 채용과제, 폐기 예정)
- 워크플로우: AC `deploy/skills/workflow/` 「채용 모드」 7-PR 분할
- 진행 세션: `PR_1_PLAN` (프로젝트 세팅 PR)
- 메인 LLM: Claude Opus 4.7
- 발생 일자: 2026-05-25

### 1차 소스 (이 문서가 인용하는 위치 — 채용 레포 폐기 후에도 의미 잃지 않도록 본문에 핵심 인용 함께 박음)

- AC `~/.claude/rules/global.md` 「조사 산출물 출처 포함」 — "외부 소스를 근거로 제시할 때 URL만 나열하지 않는다. 해당 페이지에서 근거가 되는 원문을 함께 인용한다. **1차 소스를 직접 확인한 뒤 사용한다**. 서브에이전트/Reviewer/시장 통념만 보고 표면 인용으로 끝내지 않는다."
- AC `deploy/skills/workflow/SKILL.md` 「[CRITICAL] 입력 산출물 비판적 검토」 — "세션·step 진입 시 입력 컨텍스트로 받은 **AI가 만든 결정·narrative 산출물**을 그대로 수용하지 않고 비판적으로 검토한다."
- AC `deploy/skills/workflow/SKILL.md` 「[CRITICAL] 검증 기준 = 진실 원천」 — "AI 산출물(matching 표, 산출물 md 등)을 검증 기준으로 쓰지 않는다."

## 사고 요약

PR_1_PLAN step-4 「산출물 리뷰」에서 메인 LLM이 `decisions.md`에 작성한 결정 D6 「ESLint 보강 — MP baseRules 채택, 채용과제 비범위 제외」의 stylelint 제외 근거를 다음과 같이 박았다:

> **제외**:
> - `stylelint` 연동 — 채용과제는 CSS Module/SCSS 미사용(Tailwind 단독)
> - 모노레포 전용 import 규칙

사용자가 즉시 반문: "너 혹시 채용과제라서 scss안 쓴다는건 어디 를 근거하느 ㄴ판단이야?"

검증 결과:

| 항목 | 1차 소스 확인 | 결론 |
|---|---|---|
| Tailwind 채택 | 공고「기술 스택 > 스타일링: Tailwind CSS」 명시 (1차 소스 확인 완료) | 확정 |
| **SCSS 배제** | 공고·과제요구사항·design-system.md 어디에도 명시 없음 | **메인 LLM의 비약** |
| 현재 코드 상태 | sass 미설치, `.scss` 파일 0건 | "현재 0건"일 뿐 도입 금지 근거 아님 |
| Tailwind + SCSS Module 공존 가능성 | 일반적 패턴 (Tailwind utility로 표현 어려운 케이스에 SCSS Module 보조) | 배제 근거 없음 |

즉, 메인 LLM은 BG 단계 산출물(`project.md`)에 박힌 "기술 스택: Tailwind CSS — 공고 명시 그대로" 한 줄을 보고:

1. 공고 원본을 직접 안 읽고 BG 메모만 신뢰
2. "Tailwind 채택"을 "SCSS 배제"로 비약
3. 그 비약을 PR1 의사결정 산출물(decisions.md D6)에 stylelint 제외 근거로 박음

## Root Cause

### 직접 원인

메인 LLM이 본 PR(PR1)의 의사결정 산출물에 박는 근거 문장을 **BG 산출물(AI 작성 narrative)만 보고 1차 소스(공고 원본·과제요구사항·디자인 시안)를 직접 확인하지 않은 채** 작성했다.

### 메타 원인

AC `deploy/skills/workflow/SKILL.md`「[CRITICAL] 입력 산출물 비판적 검토」가 본 사고 정확한 영역을 다룬다 — "AI가 만든 결정·narrative 산출물"을 받은 다음 step이 그대로 재사용하지 말고 비판적 검토하라는 룰. **그러나 메인 LLM은 본 룰을 적용하지 않고 BG 산출물을 그대로 인용했다**.

또한 글로벌 룰「조사 산출물 출처 포함」이 "1차 소스를 직접 확인한 뒤 사용한다"고 명시. 본 사고는 이 룰도 위반.

### 정황 원인

- BG 산출물 `project.md`는 BG 세션에서 사용자와 같이 만든 신뢰도 있는 결정 메모로 메인 LLM에게 인식되어 있다. 그러나 BG는 PR 분할·기술 스택 윤곽을 잡는 단계로, 각 항목의 1차 소스 정확 인용을 강제하지 않는다.
- 본 사고에서 BG 메모 원문은 `- TypeScript, Next.js (App Router) + React 19, Tailwind CSS — 공고 명시 그대로`. 이 한 줄을 메인 LLM이 (a) 공고 원본 안 읽고, (b) "Tailwind 명시 = SCSS 배제"로 비약, (c) 그 비약을 PR1 결정 근거로 사용 — 세 단계의 비약이 한 번에 일어남.

## 본 사고의 위치를 MP가 알아야 하는 이유

본 사고 자체는 AC workflow 스킬의 영역이지만, MP에 핸드오프하는 이유는 **MP의 코딩 컨벤션 1차 소스가 채용과제·사이드 프로젝트의 의사결정 근거로 직접 인용되기 때문**이다. 본 사고에서도 D6의 채택안 근거가 MP `docs/static-checking/eslint.md`였다 — MP 1차 소스는 정확히 인용됐지만 그 외 추가 결정(stylelint 제외)에서 1차 소스 부재 비약이 발생.

따라서 MP에 박을 룰은 두 갈래:

### (1) MP 1차 소스 인용 강도 강화

채용과제 적용 가이드(`docs/patterns/setup/ProjectSetup.md` 「채용과제 적용 시 주의」 절) 본문에 다음 명시 추가 검토:

- "본 가이드에서 채용과제용으로 **제외**하는 항목(stylelint, 모노레포 import 룰 등)은 **공고·과제요구사항·디자인 시안에서 직접 근거를 확인한 뒤 제외**한다. 채용과제의 기술 스택 명시가 다른 도구·관행을 자동 배제하지 않는다."
- 예시: "공고에 'Tailwind CSS' 명시 = Tailwind 도입 확정. SCSS/CSS Module 사용 여부는 별개 — 명시되지 않았다면 도입 가능성을 열어두고 stylelint도 보존."

### (2) MP 패턴맵 엔트리에 "1차 소스 인용 의무" 메타 룰 추가

`docs/best-practices-map.md` 상단 또는 본문에 "본 맵의 패턴을 다른 프로젝트(채용과제·사이드 프로젝트)에 적용할 때, 적용·제외 결정의 근거는 그 프로젝트의 **1차 소스**(공고, 요구사항, 디자인 시안)에서 직접 확인한 뒤 박는다. BG·사전조사 산출물은 인덱스 역할로만 신뢰한다." 같은 메타 룰 추가 검토.

본 사고 시점에 패턴맵 상단에 본 룰이 박혀 있었다면 메인 LLM이 entry 확인하면서 같이 인지했을 가능성.

## 검증 절차

PR_2_PLAN 또는 다음 채용과제에서 본 사고가 재발하는지 확인:

1. `decisions.md` 작성 시 LLM이 박는 근거 문장의 출처가 BG 산출물인지 1차 소스인지 사용자가 확인
2. 출처가 BG 산출물 한 줄에서만 도출된 경우, 1차 소스 직접 인용으로 보강했는지 점검

## 미정 항목

- 1차 소스 인용 의무를 어디까지 강제할지 — `decisions.md`에 결정 근거 적을 때마다 1차 소스 인용을 의무화하면 작성 비용 증가. 비용·정확성 트레이드오프
- MP 패턴맵 메타 룰이 너무 길어지면 entry 가독성 저하 — 별도 「적용 시 주의」 가이드 분리 검토
- AC workflow 스킬에서 「입력 산출물 비판적 검토」 룰을 어떻게 강화할지 (본 MP 문서 범위 아님 — AC 백로그로)

## 첫 행동 (이어 작업하는 사람용)

2. 위 (1)·(2) 적용 PR 작성. scope: `setting`
3. AC 측 강화는 별도 핸드오프 또는 AC 백로그로
