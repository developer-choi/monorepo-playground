# 아키텍처 결정 히스토리

## 2026-03-26: 폴더 구조 결정

### 니즈 정리

이 프로젝트는 프론트엔드 best practice를 예제로 모아두는 곳이다. 여기서 충돌하는 니즈 3가지:

1. **읽기 좋아야 한다** — 대원칙. 읽는 사람이 한 폴더만 보면 해당 예제를 이해할 수 있어야 한다.
2. **학습주제별로 독립적이어야 한다** — 각 예제는 하나의 best practice만 시연한다. 에러처리, 무한스크롤, 폼 검증 등이 뒤섞이면 개별 패턴이 안 보인다.
3. **도메인은 통일해야 한다** — 도메인(board)은 하나로 통일하되, 예제마다 구현이 달라야 한다. 같은 board인데 validation 예제의 BoardCard와 infinite-scroll의 BoardCard는 완전히 다른 컴포넌트.

추가로:

- 읽는 사람은 블로그/깃허브 링크를 통해 **학습주제 기반**으로 진입한다.
- 예제 수는 계속 늘어난다.
- 개별 예제와 별도로, 모든 best practice를 합친 **완성형 페이지(showcase)**도 필요하다.
- **중복 코드를 줄여야 한다** — 예제가 늘수록 동일한 board API 호출함수와 타입이 N벌 복사된다. 자기완결을 위해 중복을 허용하면 "이게 다 뭐지?" 해서 오히려 읽기 불편해진다.

### 양립 불가능한 두 분류 기준

**도메인 기반으로만 하면**: 비즈니스 도메인이 board 하나뿐이라 `board/` 밑에 모든 예제가 몰린다. board가 뚱뚱해지고 `board/ = src/`나 다름없어진다.

**학습주제 기반으로만 하면**: `src/validation/`만 보면 "뭘 검증한다는 건지?" 맥락이 없다. 그리고 같은 board 도메인인데 예제마다 구현이 다르니까 타입/API가 파편화된다. 어디는 에러처리, 어디는 성공처리, 어디는 UX처리 — "이걸 합치면 어떻게 되는 거지?"가 보이지 않는다.

### 1뎁스: 학습 카테고리 vs 도메인

**결론**: 학습 카테고리 (rendering, validation, showcase)

**근거**:

- 읽는 사람은 "무한스크롤 구현법"으로 진입하지, "board 관련 예제"로 진입하지 않음
- 블로그/깃허브 링크로 학습주제별 페이지를 공유하는 게 목적
- 도메인이 board 하나뿐이라 도메인 1뎁스는 의미 없음 (board/ = src/나 다름없음)

**기각된 대안**:

- 도메인 1뎁스 (`src/board/infinite-scroll/`) — 도메인이 1개뿐이라 불필요한 뎁스
- 학습주제 바로 1뎁스 (`src/infinite-scroll/`) — app 경로와 불일치

### shared에 도메인 데이터 레이어를 둘 것인가

**결론**: Board 타입, API 호출함수, DTO 변환은 shared에 둔다.

**근거**:

- 예제가 계속 늘어나는데, 동일한 board API 호출함수를 예제마다 복사하는 건 비합리적
- 중복 코드가 쫙 있으면 오히려 "이게 다 뭐지?" 해서 읽기 불편

**이전 결정과의 충돌**:

- 처음에는 "각 예제는 자기완결적, shared에 도메인 안 둔다"였음
- 이유: 읽는 사람이 한 폴더만 보면 이해할 수 있어야 해서
- 하지만 예제 수가 늘수록 중복 비용이 자기완결의 이점을 초과한다고 판단

**shared에 들어가는 것**: Board 타입, API 호출함수, DTO 변환 (모든 예제가 동일하게 쓰는 최소한)
**shared에 안 들어가는 것**: 컴포넌트, hooks, queries, zod 스키마 (예제별로 구현이 다름)

### 컴포넌트/스키마 공유 안 하는 이유

**결론**: 예제 간 코드 통합 안 함. 중복이어도 각자 구현.

**근거**:

- 코드 통합의 전제는 "사용하는 곳마다 미래에 같은 변경사항을 공유해야 하는가"
- 예제끼리는 독립적으로 변경됨. showcase의 zod 스키마를 바꿨다고 validation 예제까지 바뀌면 안 됨

### app 경로와 src 경로 1:1 매칭

**결론**: 매칭한다.

**근거**:

- 파일 찾기 쉬움. URL 보고 src 경로를 바로 유추 가능

### showcase: 완성형 페이지

**결론**: `/showcase/board` 경로로 별도 카테고리

**근거**:

- 개별 예제는 하나의 베스트프랙티스를 분리해서 보여주는 것
- 실무 페이지는 모든 베스트프랙티스의 총합. 이걸 보여주는 완성형이 필요
- 다 합치면 코드가 뒤섞여서 개별 베스트프랙티스가 안 보이므로, 개별 예제와 완성형은 공존해야 함

**네이밍**: showcase 하위 코드는 `Showcase` 접두사 사용 (예: `ShowcaseBoardDto`)
